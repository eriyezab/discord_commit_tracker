# Discord Commit Bot — Project Spec

## Overview

A Discord bot that automatically posts to a designated channel whenever someone pushes a commit to a tracked GitHub repository. The goal is passive sharing — a small friend group works on projects, and everyone naturally sees what others have been up to without anyone needing to remember to post.

The initial motivation is tracking progress on [100 Days of SwiftUI](https://www.hackingwithswift.com/100/), but the bot is repo-agnostic and supports tracking any GitHub repo.

## Architecture

GitHub Webhooks → Webhook Receiver Server → Discord Bot → Single Channel

The bot runs a lightweight HTTP server with one public endpoint. When a tracked repo receives a push, GitHub sends a POST to this endpoint. The bot verifies the request, formats the commit data, and posts an embed to a single configured Discord channel.

## Tech Stack

- **Language**: Python 3.11+
- **Discord library**: discord.py
- **HTTP server**: FastAPI (with uvicorn)
- **Database**: SQLite via aiosqlite (async-compatible)
- **Testing**: pytest with pytest-asyncio
- **Hosting**: Railway or Fly.io (must be publicly reachable for GitHub webhook delivery)
- **Dev tunneling**: ngrok (for local development)

## Concurrency Note

Both FastAPI and discord.py are async. They need to share an event loop. The recommended approach is to run the FastAPI app (via uvicorn) as a background task within the discord.py bot's event loop, or use `asyncio.gather` to run both. Avoid running them in separate threads unless necessary — a single asyncio loop is simpler and avoids thread-safety issues with the Discord client.

## Data Model

Three SQLite tables:

```sql
CREATE TABLE tracked_repos (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  guild_id TEXT NOT NULL,
  repo_full_name TEXT NOT NULL UNIQUE,
  webhook_secret TEXT NOT NULL,
  added_by TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

- `guild_id`: The Discord server ID.
- `repo_full_name`: e.g. `octocat/hello-world`.
- `webhook_secret`: A per-repo secret generated by the bot, used to verify incoming GitHub payloads.
- `added_by`: Discord user ID of whoever ran `/track`.

Second table for webhook deduplication:

```sql
CREATE TABLE webhook_deliveries (
  delivery_id TEXT PRIMARY KEY,
  received_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

- `delivery_id`: The value from the `X-GitHub-Delivery` header. Used to prevent double-posting if GitHub retries a webhook that was already processed.
- Old entries can be pruned periodically (anything older than 24 hours is safe to delete).

Third table for bot state:

```sql
CREATE TABLE bot_state (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

Used for persistent bot-wide settings. The global kill switch is stored as a row with `key = 'paused'` and `value = 'true'` or `'false'`. Defaults to `'false'` if the row doesn't exist.

The bot posts to a single hardcoded channel (configured via environment variable). No channel_id column needed.

## Environment Variables

- `DISCORD_TOKEN` — Bot token from the Discord Developer Portal.
- `DISCORD_CHANNEL_ID` — The single channel the bot posts to.
- `DISCORD_GUILD_ID` — The server ID (used for registering slash commands).
- `DATABASE_PATH` — Path to the SQLite file (default: `./data/bot.db`).
- `WEBHOOK_BASE_URL` — The public base URL of the bot's server (e.g. `https://yourbot.fly.dev`).
- `PORT` — Port for the uvicorn server (default: `3000`).
- `BOT_ADMIN_ROLE` — Name of the Discord role required to run `/track` and `/untrack` (e.g. `commit-bot-admin`). Server admins can always use these commands regardless of role.

## Discord Slash Commands

### `/track <repo>`

- **Permissions**: Requires the `BOT_ADMIN_ROLE` role or server administrator permission. Responds with an ephemeral error if the user lacks permission.
- `repo` (string, required): The full repo name in `owner/repo` format.
- Validates the format.
- Generates a unique webhook secret for this repo.
- Saves the repo to the database.
- Responds with an ephemeral message containing:
  - The webhook URL: `{WEBHOOK_BASE_URL}/github/webhook`
  - The generated webhook secret.
  - Step-by-step instructions for the user to add the webhook in their GitHub repo settings (Settings → Webhooks → Add webhook, content type `application/json`, select "Just the push event").

### `/untrack <repo>`

- **Permissions**: Requires the `BOT_ADMIN_ROLE` role or server administrator permission.
- `repo` (string, required): The full repo name in `owner/repo` format.
- Removes the repo from the database.
- Responds with a confirmation and a reminder to also delete the webhook from GitHub repo settings.

### `/list`

- No arguments.
- Responds with a list of all currently tracked repos in the server, or a message saying none are tracked.

### `/pause`

- **Permissions**: Requires the `BOT_ADMIN_ROLE` role or server administrator permission.
- No arguments.
- Sets the global `paused` flag to `true` in the `bot_state` table.
- The bot continues receiving and logging webhook events to stdout, but stops posting to Discord.
- Responds with a confirmation message in the channel (not ephemeral — the whole group should know posting is paused).

### `/resume`

- **Permissions**: Requires the `BOT_ADMIN_ROLE` role or server administrator permission.
- No arguments.
- Sets the global `paused` flag to `false` in the `bot_state` table.
- The bot resumes posting to Discord. Events received while paused are not retroactively posted.
- Responds with a confirmation message in the channel.

## Webhook Endpoint

### `POST /github/webhook`

1. Read the `X-Hub-Signature-256` header and the `X-GitHub-Delivery` header from the request.
2. Check the `X-GitHub-Delivery` ID against the `webhook_deliveries` table. If it already exists, respond `200 OK` and skip (duplicate delivery).
3. Extract `repository.full_name` from the JSON payload.
4. Look up the repo in the database to get the stored `webhook_secret`.
5. Verify the HMAC-SHA256 signature of the raw request body against the stored secret (use `hmac.compare_digest` with `hashlib.sha256`).
6. If verification fails, respond `401 Unauthorized` and drop the request.
7. If the event header (`X-GitHub-Event`) is not `push`, respond `200 OK` and ignore.
8. Filter: only process pushes to the repository's default branch. Compare `ref` in the payload against `repository.default_branch`. Ignore pushes to other branches.
9. Store the `X-GitHub-Delivery` ID in the `webhook_deliveries` table.
10. Check the global `paused` flag in the `bot_state` table. If paused, log the event to stdout and respond `200 OK` without posting to Discord.
11. For each commit in the `commits` array, post a Discord embed to the configured channel.

### Health Check

`GET /health` — Returns `200 OK`. Used for deployment health checks.

## Discord Embed Format

For each commit, post an embed with:

- **Author**: The commit author's name from the payload.
- **Title**: The commit message (first line only, truncated to 80 characters if longer).
- **Repository**: `repository.full_name`, linked to the repo URL.
- **Diff link**: Link to the commit URL (`commit.url`).
- **Private repo handling**: If `repository.private` is `true`, append a note to the embed footer: "🔒 Private repo — link may not be accessible."
- **Timestamp**: The commit timestamp.

If a single push contains multiple commits, send one embed per commit. If a push contains more than 5 commits, post the 5 most recent and add a note: "and N more commits — see full push comparison" with a link to the `compare` URL from the payload.

## Security

- Every tracked repo gets its own unique webhook secret generated by the bot (use `secrets.token_hex(32)`).
- All incoming webhook requests are verified via HMAC-SHA256 signature before processing.
- If a request's `repository.full_name` is not found in the database, respond `200 OK` silently (don't leak information about what the bot tracks).
- Slash command responses containing secrets are sent as ephemeral messages (only visible to the user who ran the command).
- `/track` and `/untrack` are restricted to users with the `BOT_ADMIN_ROLE` role or server administrator permission.

## Error Handling

- All errors are logged to stdout with enough context to debug (timestamp, error type, relevant repo or command).
- If the bot fails to post a commit embed (e.g. Discord API error, malformed payload), it logs the full error to stdout AND posts a brief, non-technical error message to the configured Discord channel (e.g. "⚠️ Failed to post commit from `owner/repo` — check bot logs for details.").
- The webhook endpoint always responds `200 OK` to GitHub even if downstream processing fails. This prevents GitHub from disabling the webhook after repeated failures.
- Slash command errors (e.g. trying to track a repo that's already tracked, or untrack one that isn't) respond with a helpful ephemeral message to the user.

## Edge Cases

### Webhook Endpoint

- **Empty push**: A push event with an empty `commits` array (e.g. branch creation or deletion). The bot should ignore these and not post anything.
- **Force pushes**: These are valid push events with rewritten history. The payload structure is the same as a normal push — no special handling needed, process as usual.
- **Push to an untracked repo**: If a repo was previously tracked and then removed via `/untrack`, the webhook on GitHub may still exist. The bot will not find the repo in the database and should silently respond `200 OK` per the existing security rule.

### Slash Commands

- **Tracking a repo that's already tracked**: `/track owner/repo` where the repo is already in the database. Respond with an ephemeral message telling the user the repo is already being tracked, and optionally re-display the webhook URL and secret in case they need it.
- **Untracking a repo that isn't tracked**: `/untrack owner/repo` where the repo is not in the database. Respond with an ephemeral message saying the repo isn't currently being tracked.
- **Malformed repo format**: Input like `just-a-name`, `owner/repo/extra`, or strings with spaces. Validate that the input matches the `owner/repo` pattern (alphanumeric, hyphens, underscores, dots, single slash) and respond with a helpful error if it doesn't.
- **Pausing when already paused**: `/pause` when the bot is already paused. Respond with a message saying the bot is already paused.
- **Resuming when not paused**: `/resume` when the bot is already active. Respond with a message saying the bot is already active.

### Discord

- **Channel unavailable**: If the configured channel is deleted or the bot loses permission to post in it, the bot should log the error to stdout and not crash. Webhook requests should still receive `200 OK` — the failure is on the Discord side, not GitHub's.
- **Embed content limits**: Discord enforces a 256-character limit on embed titles and a 4096-character limit on embed descriptions. Truncate long commit messages to fit within these limits. The spec already truncates the title to 80 characters, which is well within the limit, but the full commit message (if included in the description) needs truncation with a "..." suffix.

### Database

- **First run**: If no database file exists, the bot should create it and initialize all tables (`tracked_repos`, `webhook_deliveries`, `bot_state`) automatically on startup before doing anything else.

## Startup Behavior

- On startup, the bot logs to stdout: the number of tracked repos, the configured channel ID, the paused state, and a confirmation that the webhook server is listening.
- On startup, the bot posts a brief message to the configured Discord channel (e.g. "✅ Online — tracking N repos"). If the bot is paused, the message should indicate this (e.g. "✅ Online — tracking N repos (currently paused)").
- On startup, prune any `webhook_deliveries` entries older than 24 hours.

## Scope Boundaries

### In for v1

- Single-channel posting (configured via environment variable).
- `/track`, `/untrack`, `/list`, `/pause`, `/resume` slash commands.
- Global kill switch — pause/resume all Discord posting while still logging.
- GitHub push event webhook receiver.
- Per-repo webhook secret generation and HMAC verification.
- Webhook delivery deduplication via `X-GitHub-Delivery` header.
- Default branch filtering (ignore pushes to non-default branches).
- Private repo detection with a flag on the embed.
- SQLite persistence.
- Startup logging to stdout and a brief online message to the Discord channel.
- Deployed to a publicly reachable host.

### Out for v1

- GitHub-to-Discord user mapping (`/link`, `/unlink`) — map GitHub usernames to Discord users so commit embeds mention the right person. Requires a `user_mappings` table and changes to the embed formatting logic.
- Multi-channel support or per-repo channel routing.
- Per-repo mute/unmute (an `is_active` flag on `tracked_repos` to temporarily silence a repo without removing the webhook).
- GitHub OAuth or GitHub App for automatic webhook registration.
- Streak tracking or activity summaries.
- Commit filtering (merge commits, bot commits, etc.) — user-configurable filtering is a future feature.
- Branch selection or filtering beyond default branch.
- Support for non-push events (PRs, issues, etc.).
- Multi-server support.

## Testing Strategy

**Framework**: pytest with pytest-asyncio for async code.

### Unit Tests

**Signature verification**:
- Valid signature with a known payload and secret — should pass.
- Invalid signature (wrong secret) — should fail.
- Missing signature header — should fail.
- Tampered payload body — should fail.

**Payload parsing and filtering**:
- Standard push event with commits — extracts repo name, author, messages correctly.
- Push to default branch — processed.
- Push to non-default branch — ignored.
- Push with empty `commits` array (branch creation/deletion) — ignored, no embed produced.
- Push with more than 5 commits — only the 5 most recent are returned, with a "and N more" note.
- Private repo — flagged correctly in the parsed output.
- Force push — processed normally.

**Embed formatting**:
- Commit message truncated to 80 characters when it exceeds the limit.
- Private repo footer is present when `repository.private` is true, absent when false.
- Diff link is included correctly.
- All required fields (author, title, repo, timestamp) are present in the embed.

**Database operations** (use an in-memory SQLite database):
- Insert a tracked repo and retrieve it by `repo_full_name`.
- Insert a duplicate `repo_full_name` — should fail or be handled gracefully.
- Delete a tracked repo and confirm it's gone.
- List all tracked repos for a guild.
- Insert and look up a delivery ID for deduplication.
- Inserting a duplicate delivery ID is detected correctly.
- Pruning old delivery entries removes only entries older than 24 hours.
- Set and get the `paused` flag in `bot_state`.
- `paused` flag defaults to `false` when no row exists.

### Integration Test

**Webhook endpoint** (using FastAPI's `TestClient`, no Discord connection needed):
- POST a valid push payload with correct signature — returns `200 OK`.
- POST with an invalid signature — returns `401 Unauthorized`.
- POST with an unknown repo — returns `200 OK` (silent ignore).
- POST with a duplicate `X-GitHub-Delivery` ID — returns `200 OK` and does not re-process.
- POST with `X-GitHub-Event` set to something other than `push` — returns `200 OK` and is ignored.
- POST a valid push payload while the bot is paused — returns `200 OK`, event is logged but not posted to Discord.
- GET `/health` — returns `200 OK`.

### Manual End-to-End

- Push a commit to a tracked repo and verify the embed appears in the Discord channel.
- Run `/track`, `/untrack`, `/list` and verify the responses.
- Push to a non-default branch and verify nothing is posted.
- Run `/pause`, push a commit, and verify nothing is posted to Discord. Run `/resume`, push again, and verify the embed appears.
- Have a friend add their repo and push a commit.

### Test Fixtures

Store sample GitHub webhook payloads as JSON files in a `tests/fixtures/` directory. Include at minimum:
- A standard push event with 1-3 commits.
- A push event with more than 5 commits.
- A push event with an empty commits array.
- A push event to a non-default branch.
- A push event from a private repo.

## Future Features

### GitHub-to-Discord User Mapping (v2)

Map GitHub usernames to Discord users so that commit embeds mention the right person. When a commit comes in, the bot looks up the commit author's GitHub username and, if a mapping exists, displays a Discord mention instead of plain text.

**New slash commands**:
- `/link <github_username>` — any server member can run this to link their own GitHub account. Saves a mapping from the GitHub username to the Discord user who ran the command.
- `/unlink` — removes the calling user's mapping.

**New database table**:

```sql
CREATE TABLE user_mappings (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  guild_id TEXT NOT NULL,
  github_username TEXT NOT NULL,
  discord_user_id TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(guild_id, github_username)
);
```

**Embed change**: The author field in commit embeds should look up `commit.author.username` in the `user_mappings` table. If a mapping exists, display as a Discord mention (`<@discord_user_id>`). If not, fall back to the commit author's name as plain text.

**Edge cases to handle**: GitHub username already claimed by another Discord user (reject), user already has a mapping (update it), unlinking when no mapping exists (friendly error).

### LeetCode Submission Tracking (v2)

Track accepted LeetCode submissions for users in the server and post them to the same channel as GitHub commits. Unlike the GitHub feature (which is webhook/push-based), this is poll-based since LeetCode has no webhook support.

**External API**: [alfa-leetcode-api](https://github.com/alfaarghya/alfa-leetcode-api) — a third-party LeetCode API with a public instance at `https://alfa-leetcode-api.onrender.com/`. The relevant endpoint is `/:username/acSubmission?limit=N`, which returns the last N accepted submissions for a user including problem title, language, and timestamp.

**New slash command**: `/leetcode <username>` — registers a LeetCode username to track, linked to the Discord user who ran the command. A `/leetcode remove` command to stop tracking.

**New database table**:

```sql
CREATE TABLE tracked_leetcode_users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  guild_id TEXT NOT NULL,
  discord_user_id TEXT NOT NULL,
  leetcode_username TEXT NOT NULL UNIQUE,
  last_seen_timestamp INTEGER,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**Polling mechanism**: A background task using `discord.ext.tasks.loop` that runs on a fixed interval (e.g. every 5 minutes). On each tick, it fetches recent accepted submissions for each tracked user, compares against `last_seen_timestamp`, posts any new submissions to the channel, and updates the stored timestamp.

**Embed format**: Similar to the GitHub commit embed — problem title, difficulty, language used, and a link to the problem on LeetCode. No link to the user's actual solution since LeetCode doesn't expose that publicly.

**Open concerns**:
- The public API instance is hosted on Render's free tier and could be unreliable. A more robust approach would be self-hosting via their Docker image or hitting LeetCode's GraphQL API directly.
- Polling N users every 5 minutes means N API calls per tick. Fine for a small friend group, but needs rate-limit awareness if the user count grows.
- Posts will have up to a 5-minute delay, unlike GitHub commits which are near-instant via webhooks.

### Streak Tracking (v2)

Track consecutive days where a user has at least one commit to a tracked repo. Post milestone notifications to the channel when a user hits notable streaks (e.g. 7 days, 14 days, 30 days, etc.). Particularly relevant for challenge-based projects like 100 Days of SwiftUI.

**Implementation sketch**: A table mapping each user to their current streak count and the date of their last commit. On each incoming push event, check if the commit date is the day after the user's last recorded commit date — if so, increment the streak. If it's the same day, do nothing. If there's a gap, reset to 1. Post to the channel when the streak hits a milestone threshold.

**Open questions**: What counts as a "day" — UTC or the user's local timezone? Should streaks be per-repo or across all tracked repos for a user?

### Weekly Digest (v2/v3)

A scheduled summary posted once a week (e.g. Sunday evening) showing activity across all tracked repos. Could include: who committed to what, total commit counts per person, most active repo, and optionally a simple leaderboard.

**Implementation sketch**: A background task using `discord.ext.tasks.loop` scheduled to run once a week. It queries the stored commit data (or aggregates from the webhook_deliveries table with additional metadata) and formats a summary embed.

**Open questions**: This requires storing more commit metadata than v1 currently persists — v1 posts commits and forgets them. A new table or an expansion of the delivery log would be needed to support aggregation.

### PR and Issue Notifications (v2/v3)

Expand beyond push events to notify the channel when a pull request is opened or merged, or when an issue is opened or closed on a tracked repo. A "friend just merged a PR" notification is often more meaningful than individual commits.

**Implementation sketch**: The webhook endpoint already receives all GitHub events — it currently filters to only `push`. This feature would add handling for `pull_request` and `issues` event types. Different embed formats for each event type (e.g. PR merged gets a purple embed, issue opened gets a green one).

**Open questions**: This is opt-in per repo or global? Users might want commit notifications but not issue spam on a busy repo. Could require a filter/preference system.
